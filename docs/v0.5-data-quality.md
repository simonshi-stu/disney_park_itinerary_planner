# V0.5 Data Quality Layer

Run this before discussing or modeling wait-time data:

```powershell
git pull --ff-only
node scripts/analyze-wait-times.mjs
```

Analyze a single park date:

```powershell
node scripts/analyze-wait-times.mjs --date=2026-07-08
```

Dry run without writing generated files:

```powershell
node scripts/analyze-wait-times.mjs --no-write
```

## Outputs

- `outputs/quality_reports/YYYY-MM-DD.md`: human-readable daily report.
- `outputs/quality_reports/YYYY-MM-DD.json`: machine-readable daily report.
- `outputs/quality_reports/latest.md`: latest analyzed daily report.
- `data/processed/wait_times/wait_times_cleaned_YYYY-MM-DD.csv`: cleaned modeling table.
- `data/processed/optimizer/optimizer_ready_wait_times_YYYY-MM-DD.csv`: one selected row per snapshot, canonical attraction, and access mode.
- `data/catalog/attraction-aliases.csv`: file-based canonical attraction aliases.

## Canonical Attraction Mapping

V0.5 does not need a backend database yet. The canonical mapping is stored in a CSV catalog first:

```text
data/catalog/attraction-aliases.csv
```

The analyzer maps each source row to:

```text
canonical_attraction_id
canonical_attraction_name
canonical_category
canonical_match_source
access_mode
```

This lets preprocessing handle capitalization differences, punctuation differences, seasonal attraction names, and manually reviewed typo/alias cases without changing the raw collector.

The quality report also tracks canonical conflicts: cases where multiple source rows map to the same `canonical_attraction_id` in the same snapshot. Downstream route planning should choose the freshest eligible row instead of blindly averaging conflicting source rows.

When V0.6 introduces MySQL, this file can migrate directly into:

```text
attractions
attraction_aliases
```

Until then, adding a new alias should be a small CSV edit, not a database migration.

## Optimizer-Ready Resolver

The optimizer-ready table resolves canonical conflicts into one selected source row per:

```text
snapshot_utc + park_id + canonical_attraction_id + access_mode
```

Selection priority:

```text
1. Prefer non-stale rows.
2. Prefer usable wait-model rows over status-only rows.
3. Prefer fresher source timestamps.
4. Keep standby and Single Rider separate through access_mode.
```

For example, DCA Soarin may include historical or seasonal names such as `Soarin' Over California`, `Soarin' Around the World`, and the current `Soarin’ Across America`. They are mapped to the same DCA Soarin canonical attraction because they represent the same facility/project family for route-planning purposes, but the resolver should select the freshest eligible source row for the current snapshot.

## Core Rules

- Raw CSV files remain unchanged.
- `wait_time_minutes` keeps the original Queue Times value.
- `observed_wait_time_minutes` is the modeling target.
- Closed rows use an empty `observed_wait_time_minutes`, even when raw wait time is `0`.
- Open rows with wait `0` are kept and flagged as `open_zero`.
- Rows with stale source timestamps are flagged as `stale_source`.
- Likely entertainment rows are treated as schedule constraints, not standby wait targets.
- Possible full-day closures and temporary downtime candidates are flagged for later review.

## Single Rider Rule

Single Rider matters for solo travelers and small groups, so the data is preserved.

Single Rider rows share the same `canonical_attraction_id` as the base attraction, but use:

```text
access_mode = single_rider
```

If a Single Rider group has positive observed wait values, it can be used as:

```text
single_rider_optimizer_reference
```

If every observed Single Rider wait is `0`, it is kept only as:

```text
single_rider_availability_only
```

That means the route optimizer may know the Single Rider option exists, but should not use an all-zero wait value as a meaningful wait-time weight.

## Modeling Target

Use this field for wait forecasting:

```text
observed_wait_time_minutes
```

Use this field to filter the rows:

```text
training_eligibility
```

Initial wait-time model rows should use:

```text
training_eligibility = standby_wait_model
```

## Database Migration Note

CSV files are a V0.5 transition layer, not the final storage strategy. As the project grows, these processed tables should migrate into MySQL in V0.6:

```text
raw wait CSV -> wait_time_raw_observations
cleaned CSV -> wait_time_clean_observations
optimizer-ready CSV -> optimizer_ready_observations or API materialized view
attraction-aliases.csv -> attractions + attraction_aliases
quality reports -> data_quality_reports
```

After V0.6, CSV output can become optional export/debug artifacts instead of the primary long-term storage.
