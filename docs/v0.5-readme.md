# Version 0.5 Data Quality README

Version 0.5 turns the wait-time collector output into a cleaner data layer for future forecasting and route optimization.

The website still works as a static V0 site. V0.5 does not introduce a backend database yet. It adds a repeatable preprocessing pipeline that keeps raw data intact, creates cleaned data, resolves canonical attraction identity, and generates optimizer-ready snapshots.

## What V0.5 Adds

- Daily data quality analysis for `data/wait_times/wait_times_YYYY-MM-DD.csv`.
- Cleaned wait-time tables with quality flags.
- Canonical attraction alias mapping.
- Single Rider handling for future solo/small-group route planning.
- Canonical conflict detection and resolver output.
- Optimizer-ready CSV files with one selected row per attraction, snapshot, and access mode.
- Human-readable and machine-readable quality reports.

## Main Command

Run this after pulling the latest repository data:

```powershell
git pull --ff-only
node scripts/analyze-wait-times.mjs
```

Analyze one date only:

```powershell
node scripts/analyze-wait-times.mjs --date=2026-07-08
```

Dry run without writing files:

```powershell
node scripts/analyze-wait-times.mjs --no-write
```

## Important Data Rules

Closed attractions are not treated as real zero-minute waits.

```text
is_open = FALSE -> observed_wait_time_minutes is empty
```

Open attractions with `0` wait are kept, but flagged:

```text
quality_flags includes open_zero
```

Stale source timestamps are excluded from wait-time training:

```text
quality_flags includes stale_source
training_eligibility = exclude_stale_source
```

Entertainment, parades, and fireworks are treated as schedule constraints, not normal standby wait targets.

## Canonical Attraction Mapping

Canonical attraction means the system's standard identity for a real park item.

```text
alias = what the source API calls it
canonical attraction = what this project treats as the real item
```

Example:

```text
Soarin' Over California
Soarin' Around the World
Soarin’ Across America
```

For DCA route-planning purposes, these map into the same Soarin facility/project family:

```text
canonical_attraction_id = dca-soarin
canonical_attraction_name = Soarin'
```

The alias table lives here:

```text
data/catalog/attraction-aliases.csv
```

For Soarin specifically, this matters because `Soarin' Over California` is a historical or seasonal DCA version, while `Soarin’ Across America` is the current fresh DCA source row in the collected data. The resolver chooses the fresh non-stale row for optimizer input.

## Single Rider Rule

Single Rider data is preserved because it matters for solo travelers and small groups.

Single Rider rows share the same canonical attraction ID as the base attraction, but use:

```text
access_mode = single_rider
```

If Single Rider wait data is all `0`, it is kept only as availability information:

```text
training_eligibility = single_rider_availability_only
optimizer_wait_weight_minutes is empty
```

This prevents the route optimizer from treating all-zero Single Rider rows as meaningful wait-time weights.

## Optimizer-Ready Table

The optimizer-ready file resolves multiple source rows into one selected row per:

```text
snapshot_utc + park_id + canonical_attraction_id + access_mode
```

Selection priority:

```text
1. Prefer non-stale rows.
2. Prefer usable wait-model rows over status-only rows.
3. Prefer fresher source timestamps.
4. Keep standby and Single Rider separate.
```

This table is the best V0.5 input for future short-term forecasting and route optimization.

## Current Direction Toward V0.6

CSV files are a transition layer. They are useful for inspection, debugging, and early modeling, but they should not be the long-term storage strategy.

In V0.6, these files should migrate into a backend database:

```text
raw wait CSV -> wait_time_raw_observations
cleaned CSV -> wait_time_clean_observations
optimizer-ready CSV -> optimizer_ready_observations or API materialized view
attraction-aliases.csv -> attractions + attraction_aliases
quality reports -> data_quality_reports
```

After the database exists, CSV generation can become optional export/debug output instead of the primary storage layer.
